# 학교 행사 점수 관리 웹사이트 PRD

## 1. 개요

### 1.1 프로젝트명
Festival Manager (학교 행사 점수 관리 웹사이트)

### 1.2 목적
학교에서 열리는 행사(여러 게임/미션으로 구성)의 개인 및 반별 점수를 실시간으로 관리하고 공개하는 모바일 웹 서비스.

### 1.3 대상 사용자
- **학생**: 행사에 참여하여 점수를 획득하고 순위를 확인하는 참가자 (로그인 불필요)
- **선생님**: 각 게임/미션을 담당하며 학생들의 점수를 입력
- **관리자**: 행사/게임을 구성하고 전체 점수를 관리

### 1.4 기술 스택
- **배포 플랫폼**: Vercel
- **프레임워크**: Next.js (App Router) + React
- **스타일**: Tailwind CSS (모바일 우선 반응형)
- **데이터베이스**: Vercel Postgres (또는 Supabase / Neon) — 실시간 업데이트를 위해 폴링 혹은 Supabase Realtime 활용
- **인증**: 세션 기반 간단 인증 (쿠키 + 서버측 세션 토큰)
- **상태관리**: 서버 컴포넌트 + SWR/React Query 폴링 (2~5초 간격)

## 2. 사용자 역할 및 인증

### 2.1 학생 (비로그인)
- 로그인 없이 접속하면 기본 화면(학생 모드)이 표시됨.
- 학생은 자기 개인식별번호를 조회해 자신의 점수를 확인 가능.

### 2.2 선생님
- **아이디**: 담당하는 게임/미션의 이름 (예: `제기차기`, `퀴즈왕`)
- **비밀번호**: 관리자가 지정한 **6자리 숫자** (모든 선생님 공통)
- 같은 게임 아이디로 동시에 2대 이상의 기기에서 로그인 시도 시 → 나중 로그인은 거부되고, 관리자 화면에 경고가 발생.

### 2.3 관리자
- **아이디**: `관리자`
- **비밀번호**: 당일 날짜 6자리 숫자 (`YYMMDD`, 예: 2026-05-15 → `260515`)
  - 서버 시간 기준 자정에 자동 변경.
- 모든 행사/게임/점수/로그/설정에 접근 가능.

## 3. 데이터 모델

### 3.1 Event (행사)
| 필드 | 타입 | 설명 |
|------|------|------|
| id | UUID | PK |
| name | string | 행사 이름 (예: "2026 봄 체육대회") |
| active | boolean | 현재 활성 행사 여부 |
| created_at | timestamp | |

### 3.2 Game (게임/미션)
| 필드 | 타입 | 설명 |
|------|------|------|
| id | UUID | PK |
| event_id | UUID | FK → Event |
| name | string | 게임 이름 = 선생님 로그인 아이디 |
| created_at | timestamp | |

### 3.3 Student (학생)
| 필드 | 타입 | 설명 |
|------|------|------|
| sid | string(5) | 개인식별번호 (`GCCNN` 형식, PK) — 예: `10211` = 1학년 2반 11번 |
| grade | int | 학년 (식별번호 1자리) |
| class_no | int | 반 번호 (식별번호 2~3자리) |
| student_no | int | 반 내 번호 (식별번호 4~5자리) |

> 제약: `sid`는 5자리 문자열, 첫 자리는 1~9, 나머지는 0패딩 숫자. 반 순위 집계 키는 (grade, class_no) 조합이다 — 1학년 2반과 2학년 2반은 별도 팀.

### 3.4 Score (점수)
| 필드 | 타입 | 설명 |
|------|------|------|
| id | UUID | PK |
| event_id | UUID | FK |
| game_id | UUID | FK |
| sid | string(5) | FK → Student |
| points | int | 점수 |
| created_by | string | `teacher:<게임명>` 또는 `admin` |
| created_at | timestamp | |
| updated_at | timestamp | |

> 같은 (event, game, sid) 조합은 1건 (upsert). 이전 값은 로그로 보존.

### 3.5 ScoreLog (점수 변경 로그)
| 필드 | 타입 | 설명 |
|------|------|------|
| id | UUID | PK |
| score_id | UUID | FK → Score |
| event_id, game_id, sid | | 스냅샷 |
| old_points | int | 변경 전 점수 (최초 입력 시 null) |
| new_points | int | 변경 후 점수 |
| changed_by | string | `teacher:<게임명>` / `admin` |
| changed_at | timestamp | |

> **선생님의 최초 입력은 로그 남기지 않아도 됨 (권장: 모든 변경을 남김).**
> **관리자가 선생님 입력값을 수정할 때는 반드시 ScoreLog 기록 필수.**

### 3.6 TeacherSession (선생님 세션)
| 필드 | 타입 | 설명 |
|------|------|------|
| game_name | string | PK (같은 게임 이름으로 1개 세션만 유효) |
| session_token | string | 현재 활성 토큰 |
| device_hint | string | IP + UA 해시 (동시 로그인 경고용) |
| issued_at | timestamp | |
| last_seen_at | timestamp | |

### 3.7 Settings
| 필드 | 값 |
|------|----|
| teacher_password | 6자리 숫자 (관리자가 설정) |
| active_event_id | 현재 노출될 행사 |

### 3.8 LoginAlert (동시 로그인 경고)
| 필드 | 타입 |
|------|------|
| id | UUID |
| game_name | string |
| attempted_at | timestamp |
| resolved | boolean |

## 4. 화면 요구사항

공통:
- **모바일 우선** (iOS Safari, Android Chrome). 최소 해상도 320px 폭.
- 숫자 입력 필드는 `inputmode="numeric"` + `pattern="[0-9]*"`로 숫자 키패드 강제 노출.
- 모든 화면 상단에 현재 모드(학생/선생님/관리자) 표시.

### 4.1 학생 화면 (비로그인, 기본 화면) — 요구사항 (2), (8)
- 상단: 현재 행사 이름, 마지막 갱신 시각.
- 탭 2개:
  1. **반별 순위**: 반(01~NN)별 합계 점수 내림차순.
  2. **개인 순위**: 개인식별번호(마스킹 가능) + 점수 내림차순.
- 상단에 "내 점수 보기" 입력창: 5자리 개인식별번호 입력 → 해당 학생의 게임별 점수와 총점, 개인 순위, 반 순위 표시.
- 자동 갱신: 3~5초 폴링 또는 실시간 구독.

### 4.2 선생님 로그인 화면
- 아이디(게임 이름) + 비밀번호(6자리 숫자) 입력.
- 비밀번호 입력창은 숫자 키패드.

### 4.3 선생님 메인 화면 — 요구사항 (9)
- **상단(큰 제목)**: 로그인한 게임 이름.
- **중단**: 학생 개인식별번호 입력 (5자리, 숫자 키패드). 입력 시 해당 학년/반/번호 표시로 확인.
- **하단**: 점수 입력 (숫자 키패드) + `점수 저장` 버튼.
- 저장 시 즉시 서버 반영, 성공/실패 토스트 표시.
- 최근 입력 내역 5건을 화면 하단 목록으로 표시 (본인이 입력한 것만).
- 로그아웃 버튼.

### 4.4 관리자 로그인 화면
- 아이디 `관리자` 고정 표기, 비밀번호 6자리 숫자 입력.

### 4.5 관리자 대시보드 — 요구사항 (11)
- **경고 알림 영역 (상단 고정)**: 동시 로그인 시도 경고. 예: "⚠️ 제기차기 게임에 동시 로그인 시도 감지 (HH:MM:SS)". 확인 후 닫기.
- **행사 목록**:
  - `+ 새 행사 추가` 버튼.
  - 각 행사: 이름, 활성화 토글, **삭제 버튼**.
- **행사 상세**:
  - 게임 목록: `+ 게임 추가`, 각 게임 이름 수정/삭제.
  - 점수 편집 뷰 3가지: **게임별 / 학생별 / 반별**.
  - 점수 셀 클릭 시 인라인 편집. 저장 시 `ScoreLog` 기록.
  - 변경 이력 보기 버튼: 해당 셀의 ScoreLog 타임라인 모달.
- **설정**: 선생님 공통 비밀번호 6자리 숫자 설정.
- **학생 관리**: 개인식별번호(및 선택적으로 이름) 목록 CSV 업로드/수기 추가.

### 4.6 행사 삭제 확인 다이얼로그 — 요구사항 (12)
- "이 행사를 삭제하면 모든 게임, 점수, 로그가 영구 삭제됩니다. 계속하시겠습니까?"
- 행사 이름을 다시 타이핑해야 활성화되는 확인 버튼 (오삭제 방지).

## 5. 기능 요구사항 상세

### 5.1 개인식별번호 규칙 — 요구사항 (3)
- 형식: 5자리 문자열 `GCCNN`. 1자리 = 학년(1~9), 2~3자리 = 반(01~99), 4~5자리 = 번호(01~99).
- 첫 자리는 0으로 시작할 수 없음 (학년이 반드시 있어야 함).
- 존재하지 않는 번호 입력 시 선생님 화면에서 저장 차단하거나, 관리자가 허용한 경우 자동 등록.

### 5.2 점수 입력 흐름 (선생님) — 요구사항 (4), (9)
1. 5자리 개인식별번호 입력.
2. 점수 입력 후 저장.
3. 서버는 (event_id, game_id, sid)로 upsert. 기존 값이 있으면 ScoreLog에 기록 후 갱신.
4. 클라이언트는 입력 필드 초기화 + 다음 학생 입력 대기.

### 5.3 동시 로그인 제어 — 요구사항 (10)
- 선생님 로그인 시:
  - 기존 활성 세션이 있으면 → **나중 시도자에게 "이미 다른 기기에서 로그인 중입니다" 에러**.
  - 동시에 `LoginAlert` 레코드 생성 → 관리자 화면에서 실시간 표시.
- 관리자는 경고 화면에서 "강제 로그아웃" 버튼으로 기존 세션을 끊고 새 로그인 허용 가능.
- 세션 타임아웃: 마지막 활동 후 30분.

### 5.4 관리자 점수 편집 로그 — 요구사항 (11)
- 관리자가 점수를 변경할 때마다 ScoreLog에:
  - `old_points`, `new_points`
  - `changed_by = admin`
  - `changed_at` (서버 시간)
- 변경 이력 조회: 점수 셀 → 모달 → 시간순 리스트.

### 5.5 행사 삭제 — 요구사항 (12)
- 삭제 시 해당 Event에 속한 Game, Score, ScoreLog 모두 cascade 삭제.
- 삭제 전 이름 재입력 확인 단계.
- 삭제 로그는 관리자 감사 로그(`AuditLog`)에 별도 보존(선택).

### 5.6 비밀번호 정책 — 요구사항 (5), (6)
- 선생님 비밀번호: 관리자 설정 페이지에서 6자리 숫자로 저장 (해시 저장 권장: bcrypt).
- 관리자 비밀번호: 서버에서 `YYMMDD` 동적 계산, DB 저장하지 않음. 서버 시간대: `Asia/Seoul`.
- 로그인 실패 시 연속 5회 실패하면 해당 IP 1분 잠금 (간단 rate limit).

## 6. API 설계 (Next.js Route Handlers)

| 메서드 | 경로 | 설명 | 권한 |
|--------|------|------|------|
| GET | `/api/leaderboard` | 개인/반 순위 | 공개 |
| GET | `/api/student/[sid]` | 개인 점수 상세 | 공개 |
| POST | `/api/auth/teacher` | 선생님 로그인 | 공개 |
| POST | `/api/auth/admin` | 관리자 로그인 | 공개 |
| POST | `/api/auth/logout` | 로그아웃 | 인증 |
| POST | `/api/scores` | 점수 입력/수정 | 선생님 or 관리자 |
| GET | `/api/scores/logs` | 점수 변경 로그 | 관리자 |
| GET/POST/DELETE | `/api/events` | 행사 CRUD | 관리자 |
| GET/POST/DELETE | `/api/games` | 게임 CRUD | 관리자 |
| GET/POST | `/api/students` | 학생 CRUD | 관리자 |
| GET | `/api/alerts` | 동시 로그인 경고 목록 | 관리자 |
| POST | `/api/alerts/[id]/resolve` | 경고 해제 | 관리자 |
| POST | `/api/settings/teacher-password` | 선생님 PW 설정 | 관리자 |

## 7. 보안 요구사항
- HTTPS 강제 (Vercel 기본).
- 쿠키: `HttpOnly`, `Secure`, `SameSite=Lax`.
- 비밀번호 저장 시 bcrypt 해시.
- 관리자 API는 서버측 미들웨어에서 세션 검증.
- CSRF: Same-Origin + SameSite 쿠키로 기본 방어, 상태 변경 API에 CSRF 토큰 권장.
- 요청 레이트 리밋 (로그인/점수 입력).

---

## 8. 비기능 요구사항
- **실시간성**: 학생 순위는 5초 이내 반영.
- **동시 사용자**: 동시 접속 500명 가정 (서버리스 함수 + DB 커넥션 풀 고려).
- **모바일 성능**: 초기 로딩 2초 이내 (3G 아님, 학교 Wi-Fi 가정).
- **가용성**: 행사 당일 99% 이상.
- **백업**: 관리자 화면에서 현재 행사 데이터 CSV 내보내기.
